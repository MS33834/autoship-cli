# Why AutoShip

> Before choosing a delivery automation tool, understand how AutoShip compares to mainstream solutions (husky / pre-commit / GitHub Actions / commitizen). This page honestly marks the dimensions where AutoShip is not the strongest, so you can decide whether it fits your project.

## Comparison with Mainstream Tools

| Dimension | AutoShip | husky | pre-commit | GitHub Actions | commitizen |
|---|---|---|---|---|---|
| Local-first (offline by default) | ✅ Yes | ✅ Yes | ⚠️ Needs network to download hooks | ❌ Runs in the cloud | ✅ Yes |
| AI-generated commit message | ✅ Built-in (local/cloud model optional) | ❌ None | ❌ None | ❌ None | ⚠️ Template-based only, not AI |
| Built-in security scan (pre-commit) | ✅ Yes | ❌ None (needs hook config) | ⚠️ Relies on third-party hooks | ⚠️ Needs custom setup | ❌ None |
| Plugin extensibility | ✅ pluggy hooks | ⚠️ Git hook scripts only | ✅ Hook repos | ✅ Marketplace | ⚠️ Limited |
| Multi-language i18n docs | ✅ zh/en/ja | ❌ English only | ❌ English only | ❌ English only | ⚠️ Partial translations |
| Configuration complexity | Low (`autoship init --yes`) | Low | Medium (needs `.pre-commit-config.yaml`) | High (YAML workflows) | Medium |
| Network required | No (core commands offline) | No | Yes (pulls hooks first time) | Yes | No |
| Trigger timing | Local manual / any stage | git hook | git hook | push / PR | Manual |
| Coverage | clean + verify + commit + upload | git hooks only | Checks only | Anything (but online) | Commit convention only |

## Scenarios Where AutoShip Wins

- **You want a single command to run "clean → verify → commit → upload"**: AutoShip chains the four steps into one unified flow, no need to assemble hooks and CI yourself.
- **You care about code privacy and don't want code sent to the cloud by default**: Local-first with optional local AI models; core commands work offline.
- **You want commit messages generated by AI based on the real diff**: not just template-filling.
- **Chinese/Japanese teams**: Native i18n docs and CLI prompts lower the barrier to entry.

## Dimensions Where AutoShip Is Not Strongest (Honest Notes)

- **Ecosystem size**: husky / pre-commit have large communities and ready-made hook repos; AutoShip's plugin ecosystem is still young with fewer third-party plugins available.
- **Runtime environment**: AutoShip requires Python ≥ 3.10; husky / commitizen are friendlier to the Node ecosystem and may be smoother for pure frontend projects.
- **CI integration maturity**: GitHub Actions has a massive collection of official/community Actions and stable hosting; AutoShip's CI integration is still maturing.
- **Legacy project compatibility**: Projects with mature `.pre-commit-config.yaml` need to rewrite some hooks to migrate to AutoShip — there is a short-term migration cost.
- **Windows stability**: Core features support Windows, but some path edge cases remain. See [Known Issues](known-issues.md).

## When to Choose Other Tools

- You only need git hooks to trigger existing scripts → **husky** is lighter.
- You only need code format/lint checks and rely on a large hook repo → **pre-commit** is a better fit.
- You need complex CI matrices on PRs (multi-OS, multi-matrix) → **GitHub Actions**.
- You only want to enforce Conventional Commits without AI → **commitizen** is enough.

AutoShip is not mutually exclusive with these tools: you can use GitHub Actions in CI and AutoShip locally, or have AutoShip invoke pre-commit's checkers.
